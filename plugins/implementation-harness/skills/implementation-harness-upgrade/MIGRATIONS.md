# Harness migration ledger

Per-version record of what changed under `templates/` and **how an upgrade should apply it to an existing
`.harness/`**. The `implementation-harness-upgrade` skill reads this to explain each diff it finds and to
apply config changes additively. **Newest entry first.**

> **Maintainer rule:** every change under `templates/` MUST add/extend an entry here in the same commit ‚Äî
> see `plugins/implementation-harness/CLAUDE.md`. A missing entry means the change silently never reaches
> existing installs on upgrade.

## How the upgrade skill uses this

- **Mechanism files** (all `scripts/*`, `dashboard/*.js`, `docs/**`, `harness-CLAUDE.md`‚Üí`.harness/CLAUDE.md`,
  `README.md`‚Üí`.harness/README.md`) ‚Äî the skill content-diffs these against the reference regardless of the
  ledger; the notes here just explain *what* the diff is so the user can tell a shipped change from a local edit.
- **Project-local operational skills** (as of 1.32.0: `templates/skills/implementation-harness-*/SKILL.md` ‚Üí
  `$T/.claude/skills/implementation-harness-*/SKILL.md`, at the project ROOT, not under `.harness/`) ‚Äî same
  content-diff treatment as mechanism files, kept as a separate table since the target path differs.
- **Config/schema files** (`config/harness.env`, `config/facets.json`) ‚Äî the `ACTION:` lines are load-bearing:
  they tell the skill exactly which knobs to add (with defaults) while preserving the user's values.
- **User-data files** (`tracking/*`, `tasks/*`, `worklog/*`, `ledgers/*`, root `CLAUDE.md`/`.gitignore`/`ci.yml`)
  are never reconciled. Where a version changed one of those, it is listed under **manual attention** so the
  skill can surface it for the user to apply by hand.
- **`CHECKSUMS.jsonl`** (sibling of this file, one JSON line per released version ‚Äî `{version, files:
  {<templates/-relative path>: sha256}}`) ‚Äî the checksum fast-path: a mechanism/operational-skill file
  whose hash matches ANY past version's entry for its own path is provably never locally edited, so the
  skill auto-upgrades it with no `AskUserQuestion`. Regenerated by
  `skills/implementation-harness-upgrade/gen-checksums.sh --append`, non-negotiably, on every version
  bump (see `plugins/implementation-harness/CLAUDE.md`) ‚Äî file discovery is automatic (globs the
  mechanism directories), so no separate file list needs maintaining here.

Entry format:
```
## <old> ‚Üí <new>  ‚Äî <summary>
- mechanism: <paths + one line each>
- config: <path> ‚Äî ACTION: <exact additive instruction>
- new files: <paths the reference adds>
- renamed/removed: <old ‚Üí new | removed>
- manual attention: <user-data / root files the upgrade won't touch>
- breaking: <none | description + manual steps>
```

## 1.38.0 ‚Üí 1.39.0 ‚Äî fix review-failed re-investigation bug + human-done reviewed bug + new dashboard bucket
Two real bugs found in production use, plus a related dashboard feature, all sharing one root fix:
`tracking/reviews.json` previously meant only "a human clicked Mark reviewed on a Done task" ‚Äî extending
it to mean "anyone (human or a review-failed sweep) has actually looked at this" fixes all three.
- mechanism: `skills/implementation-harness-review-failed/SKILL.md` ‚Äî Stage 1's worklist query now
  excludes any id already recorded reviewed in `tracking/reviews.json`, so a task a past sweep already
  closed out is never re-investigated (previously EVERY `status=="failed"`/`"blocked"` task was
  re-selected on every run, forever ‚Äî this is why a real sweep found 5 tasks when only 4 needed review).
  Stage 3 now also runs one batched `mark-reviewed.sh` call covering every id the sweep investigated
  (both originally-`"failed"` and originally-`"blocked"`), which is what makes the Stage 1 exclusion
  take effect on the next sweep. Frontmatter/Stage 4 text updated to match. Note: the FIRST sweep after
  upgrading will still re-investigate every currently-failed task once ‚Äî self-healing, not a bug.
- mechanism: `dashboard/server.js` ‚Äî `/api/mark-done` now chains a `mark-reviewed.sh` call after
  `mark-done.sh` succeeds, so a human completing a needs-human task themselves no longer needs a
  separate manual "Mark reviewed" click (mirrors: marking a task failed is itself a review verdict).
  Also wires the new `failedPendingReview` bucket throughout: `loadState()`'s `counts` and its
  `buildFailures` attachment guard (fixed from `!task.failed` to `(!task.failed || !task.reviewed)` so
  the "N failed attempts" pill still shows for a failed-but-unreviewed task, not just non-failed ones),
  `renderBacklog()` (new section between Human Tasks and Done), `renderSection()`'s bulk-action-bar
  condition, `pillsFor()` (new amber "‚öÝ awaiting review" pill, distinct from Done's red "‚úó failed"),
  `renderTask()`'s "Mark reviewed" button condition (`!task.failed` ‚Üí `!task.reviewed`) and
  `showCheckbox`, and `bulkAction()`.
- mechanism: `dashboard/lib.js` ‚Äî `computeBacklog()` drops the `reviewed = isReviewed() || failed`
  auto-implication; `reviewed` now means only "has a `reviews.json` entry." A `failed && !reviewed` task
  lands in a new `failedPendingReview` bucket instead of `done`. `isStuck()` and the `done`-bucket sort
  are unaffected (confirmed by direct trace, not assumed).
- mechanism: `dashboard/lib.test.js` ‚Äî 3 tests rewritten to assert the new (opposite) invariant, 1 fixed
  by adding a `reviews` overlay to keep testing what its name says, 3 new tests added; full suite green
  (36/36).
- mechanism: `docs/HARNESS.md` ¬ß8.2, `docs/designs/manual-fail-signal.md` ‚Äî one sentence/clause each
  noting `reviews.json` is now also written by `review-failed` and the dashboard's mark-done action.
- config: none. new files: none. renamed/removed: none.
- manual attention: none.
- breaking: none. Every failure mode degrades safely (a task simply isn't excluded/auto-reviewed yet,
  never a wrong state); the one-time "first sweep re-investigates everything" transition is expected.

## 1.37.0 ‚Üí 1.38.0 ‚Äî checksum-based auto-upgrade fast path for genuinely stale (never-edited) files
The upgrade skill already refused to auto-decide on anything that wasn't byte-identical to the CURRENT
reference ‚Äî but "current" was the only version it ever checked against, so an install several releases
behind with a file that was NEVER locally edited still got a full diff captured and an `AskUserQuestion`
asked, for a file where there was, in truth, no decision to make. Fixed by widening the same trust
boundary across every past released version instead of just the latest one.
- mechanism: `skills/implementation-harness-upgrade/SKILL.md` ‚Äî Stage 0 loads the new `CHECKSUMS.jsonl`
  + a portable `sha256_of()` helper; Stage 3 hashes any file that fails `cmp -s` against the current
  reference and looks it up against `CHECKSUMS.jsonl` before treating it as "differs, needs
  adjudication" ‚Äî a match classifies it as clean/auto-upgrade instead; Stage 4 applies the auto-upgrade
  bucket directly (`cp -p`, no `AskUserQuestion`) and makes the `harness.env` missing-knob-append action
  auto-apply too (already safe by construction ‚Äî only ever adds a key the target lacks, never touches
  an existing value); Stage 1a's adoption-mode hunk classification runs the same checksum lookup first,
  so even a no-version-marker fork can fast-path a provably-unedited file; Stage 5's summary splits
  "files refreshed" into auto-upgraded vs. user-approved.
- new files: `skills/implementation-harness-upgrade/CHECKSUMS.jsonl` (the per-version hash ledger ‚Äî see
  the "How the upgrade skill uses this" note above), `skills/implementation-harness-upgrade/gen-checksums.sh`
  (maintainer-only generator, NOT shipped to `.harness/` installs ‚Äî never referenced by `create`, never
  invoked by the upgrade skill's own runtime logic). File discovery in `gen-checksums.sh` is unconditional
  (globs the mechanism directories fresh every run) ‚Äî no manifest, no exclude list, nothing to drift.
- config: none (this change alters HOW `harness.env` knob-adds are confirmed, not the file's own schema ‚Äî
  no new knob was added to `harness.env` itself by this change).
- renamed/removed: none.
- manual attention: none.
- breaking: none. Every failure mode of the checksum fast-path (a skipped `gen-checksums.sh --append`,
  a file older than the ledger's coverage) degrades to exactly today's ask-first behavior ‚Äî never a
  wrong overwrite. The checksum ledger currently covers the 5 most recent released versions
  (1.35.1 ‚Üí 1.37.0, backfilled once from this repo's own git history) plus 1.38.0 itself; an install
  older than that simply falls back to the diff+ask flow, same as before this change.

Four related fixes from a deep-dive into how failed/blocked tasks get reviewed and whether their

  `check-task-scope.sh` (previously only manually invoked) across all pending/buildable tasks and
  folds any scope-gap WARN into the GO/NO-GO verdict as a NO-GO (distinctly labeled as a heuristic

  gets a new self-check step (6a): before finalizing a draft, the agent verifies every file its own
  `specDo` text instructs touching is present in that unit's `scope` array, fixing silently if not.
  Belt-and-suspenders with the 4e check above ‚Äî this catches the gap at draft time, 4e catches it (plus
  tasks authored via other paths) at pre-flight time.
- mechanism: `scripts/mark-failed.sh` ‚Äî guard loosened to accept `status=="done"` OR
  `status=="blocked"` (previously `"done"` only), so it can also close out a reviewed blocked task, not
  just overturn a false "done" success. Header comment updated to describe both uses.
- mechanism: `skills/implementation-harness-review-failed/SKILL.md` ‚Äî the 2-way outcome split ("no
  follow-up needed" / "follow-up authored") becomes 3-way ("already resolved" / "follow-up authored" /
  "not worth pursuing"), and Stage 3 now closes out every reviewed `status=="blocked"` task via the

  tier-escalation ladder progress (`cur_rung`/`cur_attempts`) was lost on ANY process restart,
  including the exact `supervise.sh` relaunch-after-rate-limit/MAX_ITERS paths the bug depended on:
  `heartbeat_clear()` was wired into the blanket `trap ... EXIT INT TERM`, so a plain `exit 5`/`exit 4`
  wiped the heartbeat before the process actually exited. Fix: `heartbeat_clear()` now fires only at
  genuine terminal task outcomes (`block_task()`, a done-integration branch, the drained-backlog exit);

  is still pending and the heartbeat isn't older than `LOOP_HEARTBEAT_RESUME_MAX_AGE`, resumes
  `cur_rung`/`cur_attempts`/`cur_base` instead of cold-starting the ladder ‚Äî restoring only scheduling
  metadata, never a partial build diff (every attempt still rebuilds cold). `heartbeat()` now also
  writes `base` (needed to reconstruct the resumed tier). `Ctrl-C`/`SIGTERM` also preserve the heartbeat
  (consistent with the crash/rate-limit case) ‚Äî use `LOOP_IGNORE_HEARTBEAT=1` for one run to force a
  clean cold restart.
- mechanism: `docs/HARNESS.md` (¬ß3, ¬ß12), `docs/LIMITATIONS.md` ‚Äî corrected the "rung is in-memory per
  run, always cold-restarts on interruption" language to describe the new resume behavior.
- config: `config/harness.env` ‚Äî ACTION: add knob `LOOP_HEARTBEAT_RESUME_MAX_AGE` default `21600` if
  absent; add knob `LOOP_IGNORE_HEARTBEAT` default empty (unset) if absent; don't touch existing values.
- new files: none. renamed/removed: none.
- manual attention: none.
- breaking: none. `mark-failed.sh`'s loosened guard is additive (still refuses anything that isn't
  `"done"` or `"blocked"`); the heartbeat resume is a pure behavior improvement with an escape hatch.

## 1.33.1 ‚Üí 1.34.0 ‚Äî genuinely live builder/auditor output (was: one buffered dump at exit)
Diagnosed a real bug (not just a display glitch): the dashboard's "live output" tail and the builder/
auditor's plain terminal output both only ever showed content once, at the very end of a `claude -p`
invocation ‚Äî confirmed empirically (a 500-word test response sat at a flat byte count for the entire
generation, then landed in a single write the instant the process exited). Root cause: plain `-p` mode
never streams to a pipe ‚Äî it computes the whole response, then writes it once. Fixed per Anthropic's
own docs: `--output-format stream-json --include-partial-messages` (`--verbose` is mandatory alongside
it ‚Äî the CLI refuses to start without it).
- mechanism: `scripts/loop.sh` + `scripts/loop.in-place.sh` ‚Äî `run_claude()` now invokes claude with
  the streaming flags. The raw event stream goes to a NEW file, `worklog/.claude-out.jsonl` (what the
  dashboard tails live); `worklog/.claude-out` itself is reconstructed from that stream via
  `jq -Rrj 'fromjson? | select(...) | .event.delta.text'` into PLAIN TEXT and keeps its EXACT prior
  meaning and every existing consumer unchanged ‚Äî `RL_HARD_RE`/`RL_RE` rate-limit detection,
  `rl_reset_wait()`'s reset-time parsing, the audit's PASS/FAIL verdict grep, the `.audit.md` worklog
  copy. The `-R ‚Ä¶ | fromjson?` shape (not a plain `select(...)` on parsed JSON) is load-bearing: `2>&1`
  means an occasional non-JSON stderr line can land mid-stream, and naive `jq 'select(...)'` treats one
  parse error as fatal ‚Äî silently dropping every later chunk for the rest of the invocation (confirmed
  empirically with a planted bad line). Also confirmed empirically: a rate-limit phrase split across
  two streamed chunks is invisible to a per-line `grep` against the raw stream ‚Äî reconstructing plain
  text first (as above) is what makes the existing detection reliable again, not incidental.
